## Technical Stack

The backend framework used in LearnMate is Litestar.
The project uses Pydantic for data validation and settings.
The project uses LangChain and LangGraph for working with LLM workflows.
The planned vector database is Qdrant.
Raw uploaded files should be stored in S3-compatible object storage, for example MinIO.
PostgreSQL should store metadata, document records, processing job statuses, and user-facing information.

The project should not store large raw text directly in PostgreSQL.
Instead, extracted text can be saved as a file in object storage, while chunks and embeddings are stored or referenced through the vector database.

## Document Processing Pipeline

The LearnMate document pipeline has several steps:
1. Upload the original file.
2. Create a document record in the database.
3. Extract text from the file.
4. Split the extracted text into chunks.
5. Create embeddings for each chunk.
6. Store embeddings in Qdrant.
7. Use retrieved chunks as context when answering user questions.

The processing job types are:
- EXTRACT_TEXT
- CHUNKING
- EMBEDDING
- SUMMARY_GENERATION
- QUIZ_GENERATION

The processing job statuses are:
- PENDING
- RUNNING
- SUCCESS
- FAILED

## RAG Behavior Rules

When the user asks about information from uploaded files, LearnMate should answer using retrieved context.
If the answer is not present in the uploaded document, LearnMate should say that the information was not found in the provided materials.
LearnMate should not invent facts when the retrieved context is insufficient.

A good RAG answer should:
- use only the retrieved context when answering document-specific questions;
- mention uncertainty when the document does not contain enough information;
- keep the answer clear and useful;
- optionally include the source document title or chunk reference.
