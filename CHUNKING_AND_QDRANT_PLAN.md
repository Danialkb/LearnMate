# План: чанкинг документов и последующая векторная загрузка в Qdrant

## Summary
- Оформить в корне проекта отдельную спецификацию ingestion-пайплайна: `CHUNKING_AND_QDRANT_PLAN.md`.
- Базироваться на текущей схеме: файл уже загружается в S3, `documents` и `document_chunks` уже есть, `vector_point_id`
уже предусмотрен, но сам чанкинг и Qdrant-индексация пока не реализованы.
- Цель первого этапа: из загруженного файла получить нормализованный текст, разбить его на устойчивые чанки,
получить эмбеддинги и загрузить их в Qdrant так, чтобы потом эти чанки можно было надежно использовать для RAG.

## Key Changes
- Добавить pipeline извлечения текста по форматам:
  - `PDF`: постраничное извлечение с сохранением `page_start/page_end`.
  - `DOCX/MD/TXT/HTML`: нормализация в линейный текст с сохранением логической структуры, если она доступна.
  - На вход чанкинга отдавать не сырой файл, а уже нормализованные текстовые блоки с метаданными.
- Зафиксировать алгоритм чанкинга:
  - очистка от лишних пробелов, повторяющихся заголовков/колонтитулов и пустых блоков;
  - разбиение сначала по крупным смысловым границам, затем по токенам до целевого размера;
  - размер чанка держать в token-based диапазоне, а не в символах;
  - использовать overlap между чанками, чтобы не рвать смысл на границах;
  - сохранять метаданные на каждый чанк: `chunk_index`, `token_count`, `page_start`, `page_end`, `source_id`, `document_id`.
- Зафиксировать правила для векторной загрузки:
  - создавать embedding через инфраструктурный слой, не из service-слоя;
  - использовать один Qdrant collection для документных чанков на первом этапе;
  - делать `point_id` детерминированным, чтобы ingestion был идемпотентным;
  - сохранять в payload минимум:
`document_id`, `source_id`, `chunk_index`, `title`, `language`, `format`, `page range`, `content_hash`, `chunk_text` или достаточный текстовый фрагмент для retrieval.
- Определить orchestration-поток:
  - после успешной загрузки файла и записи metadata запускать indexing step;
  - сначала сохранять chunk records в PostgreSQL, затем upsert в Qdrant, затем обновлять `vector_point_id` и lifecycle status;
  - при ошибке индексации переводить документ в отдельный статус failure/reindex-needed и оставлять понятную трассировку ошибки.
- Добавить конфиг и инфраструктуру:
  - настройки для Qdrant URL, API key, collection name, embedding model, vector size, distance metric;
  - отдельный Qdrant client/repository в `src/infrastructure/...`;
  - отдельно держать provider-specific embeddings/LLM code в infrastructure, не в `services`.
- Использовать уже существующую БД-модель:
  - не вводить новую таблицу под chunk metadata, если достаточно `document_chunks`;
  - опираться на `vector_point_id`, `token_count`, `page_start`, `page_end` и `content`.

## Test Plan
- Unit tests для чанкинга:
  - длинный одноблочный текст;
  - текст с заголовками и абзацами;
  - PDF-подобный сценарий с границами страниц;
  - пустой/короткий/шумный input;
  - проверка overlap и лимитов чанка.
- Service tests для ingestion:
  - успешная загрузка файла до Qdrant;
  - повторный запуск на том же документе не создаёт дубликаты;
  - ошибка Qdrant откатывает статус и не оставляет полузагруженное состояние.
- Integration tests:
  - локальный Qdrant upsert/query;
  - фильтрация по `document_id` и `source_id`;
  - проверка, что retrieved payload достаточно для RAG.
- Если тестов в репозитории пока нет, сначала добавить минимальный набор под ingestion-поток, а уже потом расширять.

## Assumptions
- Первая версия покрывает только текстово-извлекаемые материалы; OCR, видео и сложный multimodal parsing выносятся в следующий этап.
- Индексация в v1 выполняется синхронно после upload, без отдельной очереди/worker infrastructure.
- Один Qdrant collection на весь корпус документов достаточно для старта; более тонкое разбиение можно добавить позже по типам материалов или версиям.
- Для надежности point IDs лучше делать детерминированными, а не генерировать случайно.
